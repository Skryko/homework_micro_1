## Задача 1. API Gateway

API Gateway используется как единая точка входа в систему микросервисов.  
Он принимает внешние запросы, выполняет терминацию HTTPS, проверяет аутентификационную информацию и маршрутизирует запросы к нужному внутреннему сервису.

### Сравнительная таблица решений

| Решение | Маршрутизация запросов на основе конфигурации | Проверка аутентификационной информации | Терминация HTTPS | Плюсы | Минусы |
|---|---|---|---|---|---|
| **Kong Gateway** | Да | Да, через плагины JWT, OAuth2, Key Auth, OIDC | Да | Специализированный API Gateway, много готовых плагинов, хорошо подходит для микросервисов и Kubernetes | Требует отдельной эксплуатации |
| **NGINX** | Да | Да, но часть возможностей удобнее реализуется в NGINX Plus или через внешние сервисы | Да | Очень быстрый и распространённый reverse proxy, простая базовая настройка | Open Source-версия менее удобна как полноценный API Gateway |
| **Traefik** | Да | Да, через middleware и forwardAuth | Да | Удобен для Docker и Kubernetes, хорошо работает с динамической инфраструктурой | Меньше возможностей API Gateway из коробки, чем у Kong |
| **Apache APISIX** | Да | Да, есть плагины аутентификации и авторизации | Да | Высокая производительность, динамическая конфигурация, много плагинов | Более сложная эксплуатация, требуется etcd |
| **HAProxy** | Да | Ограниченно | Да | Очень быстрый и надёжный балансировщик | Не является полноценным API Gateway, меньше возможностей для управления API |

### Выбранное решение

В качестве API Gateway предлагается использовать **Kong Gateway**.

### Обоснование выбора

**Kong Gateway** лучше всего подходит под требования задачи, потому что это специализированное решение для API Gateway, а не просто reverse proxy или балансировщик нагрузки.

Kong закрывает все необходимые требования:

- маршрутизация запросов к нужному сервису на основе конфигурации;
- проверка аутентификационной информации в запросах;
- терминация HTTPS;
- поддержка плагинов для JWT, OAuth2, Key Auth, rate limiting, логирования и мониторинга;
- возможность использования в Kubernetes и классической инфраструктуре.

Пример логической схемы работы:

```text
Client
  |
  | HTTPS
  v
Kong Gateway
  |
  |-- HTTPS termination
  |-- Authentication / Authorization
  |-- Routing
  |
  +--> user-service
  +--> order-service
  +--> payment-service
  +--> notification-service

  ## Задача 2. Брокер сообщений

Брокер сообщений используется для асинхронного взаимодействия микросервисов.
Он позволяет сервисам обмениваться событиями и сообщениями без прямой зависимости друг от друга.

### Сравнительная таблица брокеров сообщений

| Решение | Кластеризация | Хранение сообщений на диске | Скорость работы | Поддержка различных форматов сообщений | Разделение прав доступа | Простота эксплуатации | Комментарий |
|---|---|---|---|---|---|---|---|
| **RabbitMQ** | Да | Да, durable queues и quorum queues | Высокая | Да, можно передавать JSON, XML, Protobuf и другие форматы | Да, через users, vhosts и permissions | Высокая | Универсальный брокер для микросервисной архитектуры |
| **Apache Kafka** | Да | Да, сообщения хранятся в commit log на диске | Очень высокая | Да, JSON, Avro, Protobuf и другие форматы | Да, через ACL | Средняя / сложная | Хорош для event streaming и больших потоков событий |
| NATS JetStream | Да | Да, через JetStream persistence | Очень высокая | Да, payload может быть в разных форматах | Да, через subject-level permissions | Высокая | Очень быстрый и лёгкий брокер |
| ActiveMQ Artemis | Да | Да, через journal или JDBC store | Высокая | Да, AMQP, MQTT, STOMP, JMS | Да, через роли и права | Средняя | Хорошо подходит для Java/JMS-среды |
| Redis Streams | Частично | Да, через механизм persistence Redis | Высокая | Да | Ограниченно | Высокая | Подходит для простых сценариев, но не лучший вариант как основной брокер сообщений |

## Выбранное решение

В качестве брокера сообщений предлагается использовать **RabbitMQ**.

## Обоснование выбора

RabbitMQ является наиболее сбалансированным решением для типовой микросервисной архитектуры.

Он соответствует всем требованиям задачи:

- поддерживает кластеризацию для повышения отказоустойчивости;
- умеет хранить сообщения на диске в процессе доставки;
- обеспечивает высокую скорость работы;
- поддерживает передачу сообщений в разных форматах;
- позволяет разделять права доступа через пользователей, виртуальные хосты и permissions;
- проще в эксплуатации, чем Apache Kafka;
- хорошо подходит для очередей задач, событий и взаимодействия между микросервисами.

Apache Kafka лучше подходит для сценариев, где требуется event streaming, обработка больших потоков событий, аналитика и длительное хранение истории событий.
Но если основная задача — надёжный обмен сообщениями между микросервисами, маршрутизация сообщений и простая эксплуатация, то RabbitMQ является более подходящим вариантом.

Пример логической схемы работы:

```text
order-service
|
| OrderCreated event
v
RabbitMQ
|
|-- payment-queue
|-- notification-queue
|-- delivery-queue
|
+--> payment-service
+--> notification-service
+--> delivery-service
